# v0.6.0 (Planned)

Release focus: diagnosis trustworthiness and explainability.

## Day 1 Deliverables

- Freeze new report fields for quality and explainability.
- Add backward-compatible schema scaffolding in service/formatter/CRD.
- Keep runtime behavior unchanged until Day 2+ logic lands.

## New Status Fields (additive)

`DiagnosisReport.status` adds:

- `qualityScore: object`
- `uncertainties: string[]`
- `evidenceAttribution: object[]`

### Intended semantics

- `qualityScore`
  - Holds diagnosis quality metadata (for example overall score and dimensions).
  - Day 1 default: empty object `{}`.
- `uncertainties`
  - Explicit uncertainty statements for operators.
  - Day 1 default: empty list `[]`.
- `evidenceAttribution`
  - Evidence-to-source mapping (tool, timestamp, signal, objectRef).
  - Day 1 default: empty list `[]`.

## Compatibility Contract

- No breaking changes to existing fields.
- Old reports without new fields must still render and be queryable.
- API responses normalize absent fields to defaults:
  - `qualityScore = {}`
  - `uncertainties = []`
  - `evidenceAttribution = []`

## CRD Strategy

- Keep fields optional and additive under `status`.
- Use `x-kubernetes-preserve-unknown-fields` for `qualityScore` and attribution objects to allow iterative shape refinement in v0.6.x.

## Implementation Notes

- `agent/models.py`:
  - `DiagnosisResult` extended with optional fields and safe defaults.
- `agent/reporting/diagnosis_reporter.py`:
  - Always writes the 3 new status fields.
- `agent/service.py`:
  - Normalizes and exposes fields in list/detail API output.
- `deploy/base/crd.yaml`:
  - Adds schema entries for the 3 fields.
